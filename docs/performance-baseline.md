## Web Vitals telemetry

`src/lib/webVitals.ts` (native PerformanceObserver — no `web-vitals` npm dep) is wired into all three entrypoints:

- `src/main.tsx` — home page bundle reports CWV
- `src/pages/city-mount.tsx` — city pages report independently
- `src/pages/blog-mount.tsx` — blog index + posts report

Each `web_vital` event flows through the existing `track()` analytics layer (sendBeacon to `/api/events`). Metrics tracked:

- **LCP** (Largest Contentful Paint) — target ≤ 2.5s
- **CLS** (Cumulative Layout Shift) — target ≤ 0.10
- **INP** (Interaction to Next Paint) — target ≤ 200ms
- **FCP** (First Contentful Paint) — target ≤ 1.8s
- **TTFB** (Time to First Byte) — target ≤ 800ms

Each metric reports ONCE per page session, fired on `pagehide` / `visibilitychange`-to-hidden so we capture the final value.

## Critical-path optimisations (Tier 3 wave)

1. **Lazy-loaded below-fold sections** (`App.tsx`):
   - `BeforeAfterGallery`, `CredentialsWall`, `FAQ`, `BookingWidget`
   - Each becomes its own JS+CSS chunk, dropped from the main bundle
   - Saves ~13 kB JS + ~14 kB CSS on initial load
   - Sized Suspense fallbacks (`<DeferredFallback height={N} />`) preserve layout to avoid CLS while chunks hydrate

2. **Critical-asset preload** (`index.html`):
   - `/images/house-1.jpg` (hero video poster, LCP-relevant)
   - `/logo-mark.webp` (hero crest, above-the-fold)
   - `fetchpriority="high"` on both
   - Hero video itself NOT preloaded (11 MB — autoplay default prioritisation is sufficient)

3. **Font loading** — Google Fonts URL on every HTML entry includes `&display=swap`. Verified across `index.html`, all 4 city HTMLs, `blog.html`, and the per-post HTMLs generated by `scripts/build-blog-routes.mjs`. No FOIT issues expected.

4. **Preconnect** — `<link rel="preconnect">` to `fonts.googleapis.com` and `fonts.gstatic.com` (with `crossorigin`) on all HTML entries. Cuts ~100ms off first-font-paint on cold visits.

## Image optimisation backlog

Audit of `public/images/*.jpg` reveals files that exceed our 300 kB-per-image budget:

| File | Size | Status |
| --- | --- | --- |
| `house-1.jpg` | 287 kB | OK (just under budget) |
| `house-2.jpg` | 850 kB | **OVER** — needs WebP conversion + quality optimisation |
| `house-3.jpg` | 460 kB | **OVER** |
| `house-4.jpg` | 429 kB | **OVER** |
| `house-5.jpg` | 308 kB | OVER (marginal) |
| `tree-oak-canopy.png` | 530 kB | **OVER** — png in particular is expensive |

These images appear in the `ServiceFeatures` section (`house-2` through `house-5`) and the `WindUpliftVisualizer` scenes. Converting to WebP at quality 82 would typically reduce size by 50-70% with imperceptible visual difference.

**Action item (deferred — separate phase):** run a Sharp pipeline on these (similar to `scripts/optimize-logos.mjs`) to produce both `.webp` and a smaller `.jpg` fallback. Update `<img srcset>` in components to prefer the WebP. Estimated impact: ~1.5 MB saved on cold home-page visit.
